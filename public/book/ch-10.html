Chapter 10 — The Breach That Wasn't

Chapter 10 — The Breach That Wasn't

The report arrived on a Tuesday in July, in the Stonethrow queue, between a duplicate XSS and a complaint about markdown rendering. Severity: critical. Author: @sable. Title: "Your guard is posted at a door you no longer use."

Marisol read it twice, then escalated. By ten it was on the war-room screen and nobody was talking.

Any authenticated researcher token can GET and POST to /api/v2/admin/programs/{id}/payout-settings. I changed the bounty multiplier on a program I do not own. I changed it back. Reproduction below. Separately: your API reflects Access-Control-Allow-Origin from whatever Origin I send, with Access-Control-Allow-Credentials: true. A page I host can read authenticated responses from your API in any logged-in researcher's browser. — s.

Lena set her coffee down carefully. "So our best researcher just told us that on our security platform, a researcher can rewrite the payout math for a bank's program. And then read other people's data from a website she controls."

"It's valid," Maya said. She'd already run the reproduction. Her own JWT — a plain researcher token, no admin role — had sailed straight into the admin endpoint and come back 200 OK with Northbeam's payout configuration.

Idris was three terminals deep in access logs. "Give me twenty minutes. I want to know if anyone else found it first."

Elias was at the Wall, at the Request Journey diagram born in March, the week Sable found the leaking stack traces. He picked up the green pen and drew a thick vertical line in front of the DispatcherServlet box, then a row of small boxes in front of the line — before any controller, before the front door Maya had spent the spring rebuilding. The stretch he was drawing on was the one he'd tapped in March and left deliberately empty — out past the filters, where the same reporter's floor-plan leak had proved that what happens in front of the servlet never reaches a @ControllerAdvice. Same reporter, bigger stakes; the geography lesson had been a forecast.

"Security lives in front of the front door," he said. "I told you in March that chapter ten of your year would happen out here. Welcome to it."

The investigation

Maya pulled up the two suspects. The first was the route guard.

⏸ Your call. A signed, unexpired researcher token came back 200 OK from an admin route — no exception, no error log, nothing visibly broke. Before the code loads, commit: what class of bug lets a request through without anyone deciding to allow it — and did it get there by error, or by drift?

@Bean
SecurityFilterChain api(HttpSecurity http) throws Exception {
    return http
        .csrf(csrf -> csrf.disable())
        .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
        .authorizeHttpRequests(a -> a
            .requestMatchers("/api/admin/**").hasRole("ADMIN")   // ← still guarding v1
            .anyRequest().authenticated())
        .oauth2ResourceServer(o -> o.jwt(Customizer.withDefaults()))
        .build();
}

She felt the bug before she could name it. At Meridian the check would have been the first line of the handler itself — an if asking who you were before the method did anything; the guard traveled with the code it guarded. Here the rule lived elsewhere. It sat in a filter, matching a URL string. And the URL had moved out from under it.

"March," she said slowly. "When I versioned the public API. I moved the admin controllers under /api/v2/admin/**. The matcher still says /api/admin/**. So a request to /api/v2/admin/programs/{id}/payout-settings matches no specific rule at all. It falls through to anyRequest().authenticated() — and any researcher with a valid token is authenticated."

"Authenticated," Elias said. "Not authorized."

"Right." She wrote it on the Wall under his vertical line. Authn ≠ Authz. "The token's real — signed, unexpired, a genuine Glasshouse researcher. Authentication asks who are you, and the answer was honest. Authorization asks are you allowed — and nobody asked, because the only rule that would have asked was guarding a door we stopped using in March."

(If you called drift at the pause: exactly — nothing malfunctioned; the chain enforced precisely what it said, and what it said had stopped describing the system. Hold that distinction; Elias prices it at the kitchen, later that afternoon.)

The second suspect was older and uglier: a config class with a 2021 commit and a comment that read // unblock the SPA team.

@Bean
CorsConfigurationSource corsConfigurationSource() {
    var cfg = new CorsConfiguration();
    cfg.setAllowedOriginPatterns(List.of("*"));     // reflects ANY origin
    cfg.setAllowCredentials(true);                  // ...with credentials
    cfg.setAllowedMethods(List.of("*"));
    var source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", cfg);
    return source;
}

"Whoever wrote this," Maya said, "thought CORS was an access-control list."

Idris looked up from the logs. "It isn't?"

"No. CORS is a browser contract. For anything non-trivial the browser sends a pre-flight OPTIONS first — 'may a page from this origin send a POST with an Authorization header?' — and on the real response it reads Access-Control-Allow-Origin to decide whether the JavaScript that asked ever sees the body. The server blocks nothing; it only tells the browser what to permit." She tapped the screen. "Reflecting every origin with credentials tells every browser: let any website read authenticated Glasshouse responses. It's not a lock we left open. It's a sign on the door saying 'all welcome, bring your keys.'"

"No exploitation," Idris announced, leaning back. "Six months of logs. Nobody reached /api/v2/admin/programs/*/payout-settings with a non-admin token except Sable last night — and you, eleven minutes ago. It's a breach that wasn't. She found it first."

The room exhaled. Then the second conversation started — the one Maya would remember longer.

"We fix this quietly," someone from the exec side said. "Patch the matcher, rotate, move on."

Lena was already shaking her head, and so was Elias. "We pay this one in full and we say thank you in public," Lena said. "A security platform that hides its own holes is a punchline waiting for the first researcher who doesn't report what they find. Trust is the product. The glass house only works if we let people see in."

She won. She usually won when she was right about money, and trust was money.

What the senior sees

That afternoon Elias caught Maya by the kitchen, away from the room.

"Your plan. Change /api/admin/** to /api/v2/admin/** and ship."

"It would work," she said.

"You're fixing the hole. Fix the class of hole." He uncapped the green pen. "Next quarter someone adds /api/v3, or a new admin controller nobody updates a matcher for, and the silent fall-through to authenticated() opens again. The model is wrong, not the string."

Her Meridian instinct was no shelter either. Scattered per-handler if checks meant a forgotten if was a forgotten check — fail-open by omission, the same trap in different syntax. So the bug isn't the typo. The bug is that the default is 'allow.'

"Deny by default," she said. "Make the drift structurally impossible instead of carefully avoided."

"Good. And while you're in there — tell me what actually validates a token. Not 'Spring does it.' Show me the mechanism."

She walked it. "The Authorization: Bearer header gets pulled by the BearerTokenAuthenticationFilter — one of the boxes on your line, early in the chain with the other authentication filters. It hands the token to a JwtDecoder — Nimbus under the hood — which verifies the signature against the issuer's JWK set, cached so we're not calling out per request. Then the claim checks: issuer is who we trust, expiry hasn't passed — and audience is us and not some other service sharing the auth server, which isn't automatic: signature, issuer, and expiry come as defaults, but the expected audience we have to declare ourselves — the resource-server audiences property, or our own validator. Only then does a JwtAuthenticationConverter map the claims to authorities — our roles claim becomes ROLE_*, which is the only reason hasRole means anything — and the Authentication lands in the SecurityContextHolder, a ThreadLocal. At the other end, the AuthorizationFilter enforcing the URL rules runs last in the chain, once authentication has had its chance. Authentication first, authorization last. The order is the architecture."

"And if the signature's forged?"

"The decoder throws, the filter's entry point answers 401, and the request dies in the filter chain. The controller never runs. That's the whole point of security living in filters — a bad request never reaches the servlet."

"One more box that isn't on your line," Elias said. "We validate the token. Who mints it?"

"Not us — and that's the half of OAuth2 people skip." The token check she'd once hand-rolled in a servlet filter at Meridian had validated tokens the same codebase minted — issuer and validator were one program, so the split never had a name. "OAuth2 is a protocol with two roles. The authorization server — Keycloak, in our cluster — is the thing that actually authenticates a person or a service and issues the token. We're the resource server: we never see a password, we only validate and enforce. How the token gets issued is a flow — authorization-code with PKCE for humans in browsers, client-credentials for service-to-service — and the old implicit and password grants are deprecated because they leak credentials into places they don't belong. The JWT is just the token's format. OAuth2 is the contract about who mints it and who trusts it — which is why the DSL says oauth2ResourceServer and not parseJwt: it's naming our role in the protocol, not the parsing."

"Good. Now the question Priya's going to ask you at standup tomorrow." Elias almost smiled. "Why is CSRF off?"

The fix

She rebuilt it from the filter chain up and inverted the model.

@Bean
SecurityFilterChain api(HttpSecurity http) throws Exception {
    return http
        .cors(Customizer.withDefaults())                  // explicit allow-list bean below
        .csrf(csrf -> csrf.disable())                     // ADR-010: no cookie credential
        .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
        .authorizeHttpRequests(a -> a
            .requestMatchers("/actuator/health").permitAll()
            .requestMatchers("/api/v2/admin/**").hasRole("ADMIN")
            .requestMatchers(HttpMethod.GET, "/api/*/reports/**").authenticated()
            .anyRequest().denyAll())                      // deny-by-default: silence means 403
        .oauth2ResourceServer(o -> o.jwt(j ->
            j.jwtAuthenticationConverter(jwtAuthenticationConverter())))
        .build();
}

anyRequest().denyAll() did the real work. A new route was unreachable until someone wrote an explicit rule for it, so a forgotten path failed closed — 403, not 200. The matcher couldn't drift open; there was no open to drift to.

The kitchen walkthrough had named the one claim check the framework wouldn't assume. The diff that delivered it was two lines of YAML.

spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://auth.glasshouse.io/realms/glasshouse
          audiences: atrium   # ADR-010: the one claim validation the framework
                              # will NOT assume — a valid token minted for another
                              # service must bounce off this line

And hasRole("ADMIN") was only a real rule because the chain no longer ended in Customizer.withDefaults(). Out of the box the converter maps the scope/scp claim to SCOPE_* authorities and nothing else. The default speaks SCOPE_, not ROLE_. Keycloak's realm roles live in their own claim. The mapping was hers to declare, like the audience.

@Bean
JwtAuthenticationConverter jwtAuthenticationConverter() {
    var roles = new JwtGrantedAuthoritiesConverter();
    roles.setAuthoritiesClaimName("roles");   // Keycloak realm roles, flattened into the token
    roles.setAuthorityPrefix("ROLE_");        // the default maps scope -> SCOPE_*, never ROLE_*
    var conv = new JwtAuthenticationConverter();
    conv.setJwtGrantedAuthoritiesConverter(roles);
    return conv;
}

But URL rules were coarse. "Any researcher can GET reports" wasn't the rule. The rule was "a researcher reads their own reports; a triager transitions status; an admin manages programs." That belonged on the domain methods, as defense in depth. Records at the boundary, not entities.

@Service
public class ReportService {

    @PreAuthorize("hasRole('RESEARCHER') and @reportAccess.owns(#reportId, authentication)")
    public ReportView read(long reportId) { /* load, map to record */ }

    @PreAuthorize("hasRole('TRIAGER')")
    public void transition(long reportId, Status next) { /* ... */ }

    @PreAuthorize("hasRole('ADMIN')")
    public void updatePayoutSettings(long programId, PayoutSettingsUpdate update) { /* ... */ }
}

@reportAccess was a small bean SpEL could call by name. The ownership logic lived in one tested place instead of copy-pasted strings.

She paused at @PreAuthorize and felt February all over again — the smoke-cloud drawing. "Method security is an AOP proxy," she told Priya, who'd wandered over. "Same proxy as @Transactional. If ReportService calls its own updatePayoutSettings through this., the proxy never sees the call and the check evaporates." The February discipline already had it covered — ADR-003 had made this.-invoked annotated methods a review finding, so @PreAuthorize simply joined the list the smoke cloud predicted. She made a note for the tests anyway: assert the rule fires on the injected bean, never on a hand-constructed one.

The CORS fix was the smallest diff and the biggest mindset change: ACL out, named origins in.

@Bean
CorsConfigurationSource corsConfigurationSource() {
    var cfg = new CorsConfiguration();
    cfg.setAllowedOrigins(List.of("https://app.glasshouse.io"));   // named, not reflected
    cfg.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
    cfg.setAllowedHeaders(List.of("Authorization", "Content-Type"));
    cfg.setAllowCredentials(true);                                  // safe now: origin is fixed
    var source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", cfg);
    return source;
}

Then CSRF — Priya's standup question, exactly as Elias predicted.

"Because JWT," Priya said.

Elias waited. He was good at waiting. Maya let the silence work, then nudged. "Go a layer down. Why does a bearer token make CSRF moot?"

Priya thought. "CSRF works because... the browser sends the cookie automatically? On any request to our domain, even from an attacker's page?"

"That's it," Maya said. "CSRF exploits ambient credentials — a session cookie the browser attaches to every request to our origin whether the user meant to send it or not. The attacker's page can't read the response, but it can cause a state-changing request, and the cookie rides along for free — hence the CSRF token, a value that page can't obtain. But our credential lives in the Authorization header — JavaScript has to attach it deliberately; the browser never sends it ambiently. No ambient credential, no CSRF surface — leaving it on would guard nothing and break our own legitimate POSTs." She wrote the rule on the Wall: cookie session → CSRF on; pure token API → CSRF off. "Same logic kills session fixation: in the cookie world an attacker pre-plants a session ID and waits for the victim to log into it — why Spring Security rotates the ID at login. No session, nothing to fixate. The skill isn't 'JWT is safer.' The skill is knowing which credential is ambient and defending that." The ADR recorded the rest: off because no cookie credential; revisit the day a cookie appears.

Before the Wall session broke up, Elias handed her the green pen. "One more drawing. Not the fix — the architecture. Walk the request again, but this time don't tell me what each layer does. Tell me what each layer is allowed to trust from the one before it."

She started at the far left, where the Wall met the window.

"Zero standing trust," she said, stepping back. "Identity is cryptographic or absent, and silence past the matchers is a 403. Each layer re-checks instead of trusting the one upstream: the chain settles who; the method layer re-asks whether — role and ownership — even though the request is already in. The chain authenticates, the method authorizes, the data layer assumes nothing."

"And the arrows. What crosses them?"

"Claims. Every arrow is a token boundary, and audience is the contract between zones — the claim naming which boundary the token was minted to cross. A token Keycloak issues for some other service carries a perfectly valid signature; it bounces off Atrium only because the audience check is ours to declare. Which is why it isn't a default."

"Now say the quiet part. What failed in July — the matcher, or the model?"

"The model. Fail-open is never designed; it accumulates — every refactor that outruns a URL string widens it, and nobody approves the purchase because nobody sees a diff. Deny-by-default makes fail-closed a structural property instead of a maintained one. That's the -ility on this wall — security as architecture, not as vigilance: making the safe thing the lazy thing." She wrote that last sentence under the zones, and then, before it cooled, the half-page:

ADR-010 — Security architecture. Context: @sable filed a valid critical — an admin route reachable without a role check, plus a credentialed wildcard CORS policy; the URL rules had drifted from the routes they guarded. Decision: Atrium is a stateless OAuth2 resource server (Keycloak issues, Atrium validates — signature, issuer, declared audience, expiry); the chain ends in anyRequest().denyAll(); method security adds role-plus-ownership rules as defense in depth; CSRF stays off for the token API on the record — no ambient cookie credential — and CORS is a named allow-list; access tokens run short with refresh rotation. Consequences: A forgotten route now fails closed (403) instead of open, at the price of an explicit rule for every new route — and a leaked token stays valid until expiry, a trade-off accepted in writing rather than discovered in an incident.

Afterward, in the corridor, Elias said quietly: "That. That's the job. Consider this your notice — months before anyone else gets theirs." And he told her — privately, long before any all-hands would — that he was stepping back at year's end, and that this rebuild had been hers for a reason.

The password path was boring in the best way: DelegatingPasswordEncoder, BCrypt with the {bcrypt} prefix at strength 12. "If this had been plaintext, or anything reversible," she told Priya, "the whole report would be a footnote — the storage itself would be the breach. BCrypt is deliberately slow, and the strength is a log scale: 12 means 2¹² rounds, so each increment doubles the cost. Milliseconds per login for us; an economic dead end for whoever steals the dump. And the prefix means we can migrate to Argon2 later without invalidating a single existing hash."

The validation checklist went up on the Wall next to the filter boxes — signature → issuer → audience → expiry → SecurityContext — with the uncomfortable last line: a stolen token stays valid until it expires. Statelessness cuts both ways.

⚖ Your move, architect

That last line is a fork, and ADR-010 had to take a side. Sable's next report, in some version of the future, is a leaked researcher token. Before reading what they chose, commit to a revocation posture:

  • A — Short TTL + refresh rotation, nothing else. Access tokens live minutes; refresh tokens rotate on every use, and a replayed refresh token kills the whole family. The exposure window is priced and accepted.
  • B — A denylist on the hot path. Revoked token IDs (jti) in a shared store, checked on every request. Real revocation, today.
  • C — Opaque tokens + introspection. The resource server asks the authorization server about every token it sees; Keycloak becomes the per-request source of truth.

The trade-offs, since you've committed: B gets real revocation but gives up statelessness — the shared store joins every request's critical path, adding latency and a new failure mode: when the denylist store is down, do you fail open or closed? You've just rebuilt the question this chapter answered. C gives up availability: the auth server joins every request's critical path — April's Cortexa lesson, where one synchronous dependency's bad afternoon spent the whole thread pool, the coupling ADR-006 taught the team to price. A is what they chose and what ADR-010 records: a bounded exposure window — fifteen minutes, knowable, in writing — in exchange for the horizontal scaling that justified JWTs in the first place. Every option sells an -ility; the architect's answer is naming which. The mature answer is never "you can't revoke a JWT." It's "we decided not to, here is the price, and here is the receipt."

They shortened access tokens to fifteen minutes with refresh rotation, and recorded the heavier options — a denylist of revoked token IDs, live introspection — as the day-two path, each buying back control by selling statelessness.

And one more mandate, hers: a security test per route — 401 without a token, 403 with the wrong role, 200 with the right one. The thing that would have caught the drift the day it shipped. Idris read the plan over her shoulder and said, mildly, "That's a few hundred new test classes on a CI already creeping past twenty minutes." Nobody heard him. They would, in August.

Sable's retest came back two days later, one line: "Clean. And thank you for publishing. — s."

The postmortem went out publicly with Maya's name on it — the matcher, the CORS bean, the rebuild, the bounty paid in full. She left July carrying two things: a security model anyone could explain from the Wall, and a date in December nobody else knew yet.

The debrief — Ch.10, The Breach That Wasn't

JWT stateless auth — where in the filter chain is the token validated, and why is CSRF protection usually disabled?

Claim: In a stateless JWT API the token is validated in the security filter chain — before the DispatcherServlet ever runs — and CSRF protection is disabled because there is no ambient cookie credential to forge. Mechanism: The BearerTokenAuthenticationFilter extracts the Authorization: Bearer header and hands it to a JwtDecoder, which verifies the signature against the issuer's cached JWK set, then checks issuer, expiry, and (opt-in) audience; on success a converter maps claims to authorities and populates the SecurityContextHolder, and the AuthorizationFilter at the end of the chain enforces the access rules. Browsers auto-attach session cookies cross-site — that is CSRF; a header token is never sent ambiently, so a CSRF token would guard nothing. Trade-off: Stateless tokens scale horizontally with no session store, but you surrender server-side revocation — the token is good until expiry no matter what you learn afterward. Failure mode: A leaked token stays valid for its full TTL — the JWT revocation problem. Mitigate with short TTLs plus refresh rotation, a per-request denylist of token IDs, or live introspection — each trading statelessness back for control.

THE ARCHITECT'S LENS

  • -ilities at stake: Security as architecture, not vigilance: trust zones with explicit token boundaries, and fail-closed as a structural property (denyAll()) versus fail-open as accumulated drift — the hole was opened by evolvability (a March refactor outrunning a URL string), which is why the fix is a model, not a matcher. Scalability is what statelessness buys; the revocation window is what it costs, priced in writing. Idris's unheard remark is operability debt accruing, due in August.
  • The ADR: ADR-010 — security architecture: stateless OAuth2 resource server (Keycloak issues, Atrium validates — signature, issuer, declared audience, expiry), anyRequest().denyAll() at the chain, method security with ownership rules as defense in depth, CSRF off on the record, CORS as a named allow-list, fifteen-minute tokens with refresh rotation.
  • Rejected alternatives: patching the matcher string and shipping (fixes the hole, not the class of hole — the drift re-opens at /api/v3); per-route annotations as the only layer (fail-open by omission — the same drift in annotation syntax); a denylist or introspection today (each buys revocation back by selling statelessness or availability — recorded as the purchasable day-two path instead).
  • Fight or lean: Lean on the resource-server machinery — BearerTokenAuthenticationFilter, the Nimbus decoder, the cached JWK set; hand-rolled token validation is how signature-bypass CVEs are born. Fight the permissive defaults you wrote yourselves — the wildcard CORS bean and the authenticated() fall-through were Glasshouse's ink, not Spring's — and remember the two checks the framework will not assume for you: the audience, and denyAll() itself.

Story anchors

  • The matcher guarding /api/admin/** while the routes moved to /api/v2/admin/** → URL-string rules drift; anyRequest().denyAll() makes a forgotten route fail closed instead of open.
  • The CORS bean reflecting any origin with credentials, "to unblock the SPA" → CORS is a browser contract, not a server ACL; name your trusted origins.
  • Sable's researcher token coming back 200 OK from the admin endpoint → authentication (who you are) is not authorization (what you may do).
  • The trust-zone map on the Wall — "the chain authenticates, the method authorizes, the data layer assumes nothing" → zero-trust posture: each layer re-checks instead of trusting upstream, and the audience claim is the contract that says which boundary a token was minted to cross.

Interview ammo

  • JWT stateless auth — where in the filter chain is the token validated, and why is CSRF usually disabled? → In the BearerTokenAuthenticationFilter, before the servlet — signature, issuer, expiry, opt-in audience, then SecurityContext; CSRF is off because a header token isn't ambient the way a cookie is — it goes back on the day a cookie appears.
  • Your anyRequest().authenticated() let a researcher reach an admin route — what's the real fix? → Authenticated isn't authorized: invert to denyAll() with explicit allows so unknown routes fail closed, then layer @PreAuthorize ownership rules as defense in depth — remembering method security rides the same self-invocation-prone AOP proxy as @Transactional.
  • How do you revoke a JWT before it expires? → You don't, by design — you pick a poison: short TTL plus refresh rotation (cheap, small exposure window), a denylist of token IDs (a per-request lookup), or introspection (full control, fully stateful again).
  • Architect follow-up — Your tokens can't be revoked; defend that to a security review. → Name the sale and show the receipt: ADR-010 prices a bounded fifteen-minute exposure window (refresh rotation with reuse detection killing the family) against the statelessness that lets the API scale with no shared store on the hot path — and records denylist and introspection as purchasable day-two options, so "can't revoke" becomes "decided not to, in writing."
